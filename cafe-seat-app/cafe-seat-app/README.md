# カフェ空席管理システム (Cafe Seat Manager)

Firebaseを活用し、店側とお客様側をリアルタイムでつなぐ座席管理Webアプリです。
単なる機能実装だけでなく、実務での利用を想定したUX（ユーザー体験）とデータの整合性にこだわって開発しています。

---

## 🛠 実装済みの主要機能

### 1. リアルタイム・フル同期
- **Firestore (onSnapshot)** を採用。
- 店側がステータスを変更（空席 ↔ 利用中 ↔ 予約済）した瞬間に、お客様側の画面へ**リロードなしで即座に**反映される双方向同期を実現しました。

### 2. データの整合性保護（バリデーション）
- **二重登録防止**: `getDoc` と `exists()` を組み合わせ、既に存在する席名が入力された場合にアラートを表示し、不用意なデータ上書きを阻止するロジックを実装。

### 3. オペレーションに最適化したUX
- **スピード入力**: 席名入力後、マウスを使わずに `Enter` キーだけで座席を追加可能。
- **入店ショートカット**: 予約客の来店時に、1クリックで「利用中」へ移行できる専用ボタンを配置し、接客フローを簡略化。

## 📈 学習の記録と課題解決 (Troubleshooting)

開発の過程で直面した技術的な壁と、その解決プロセスを記録しています。

### 🔴 課題：修正コードが反映されない「キャッシュ」問題
- **事象**: 二重登録防止ロジック（`getDoc`）を正しく実装したにもかかわらず、ブラウザ上で古い挙動（上書き）が続いてしまった。
- **原因の切り分け**: 
  1. `console.log` を差し込み、バリデーション関数自体が呼ばれているか確認。
  2. ログが出力されないことから、ブラウザが古いJavaScriptファイルをキャッシュ（保持）し続けていると推測。
- **解決策**: ブラウザの強制リロード（Ctrl+F5）およびLive Serverの再起動を実施。
- **学び**: 挙動がロジックと矛盾する場合、コードのミスだけでなく**「実行環境のキャッシュ」**という外部要因を疑い、環境をクリーンにする手順の重要性を学んだ。

### 🔴 課題：ローカルからクラウドへのデータ移行
- **事象**: `localStorage` では複数端末（店員と客）間でのデータ同期が不可能だった。
- **解決策**: Firebase (Cloud Firestore) を導入。データを「ブラウザ単位」ではなく「クラウド一元管理」へ再設計し、リアルタイム性を確保。
